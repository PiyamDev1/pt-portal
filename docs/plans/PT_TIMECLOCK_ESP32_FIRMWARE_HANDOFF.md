## Current firmware gaps

The QR payload generated by `src/services/service_qr.cpp` now matches the Portal
contract. Do not change its signing material.

The remaining service integration is not active yet:

1. `service_http_tick()` returns immediately when `state.provisioning_complete`
   is true.
2. `http_get_json()` and `http_post_json()` do not add signed device headers.
3. Config, heartbeat, and notice handlers exist but are never called for a
   provisioned PT Portal device.
4. No handler exists for `POST /api/timeclock/devices/manual-code`.
5. `ui_qr.cpp` creates a manual-code label but leaves it set to `Unavailable`.
6. `WiFiClientSecure::setInsecure()` disables TLS certificate validation.
7. Existing HTTP helpers run synchronously and can stall the LVGL loop.

## Device request authentication

Every request to the four Portal service routes must include:

```text
X-PTC-Device-Id: <device UUID>
X-PTC-Timestamp: <current Unix seconds>
X-PTC-Nonce: <new random base64url value>
X-PTC-Signature: <base64url HMAC-SHA256>
```

Generate a fresh nonce for every HTTP attempt. Never reuse a nonce when
retrying a request. A 16-byte random value encoded as unpadded base64url is
appropriate.

The signature input is exactly five lines:

```text
<UPPERCASE HTTP method>
<pathname and query>
<Unix timestamp seconds>
<nonce>
<lowercase SHA-256 hex of the exact request body bytes>
```

Equivalent construction:

```cpp
String material = method + "\n" +
                  path_and_query + "\n" +
                  timestamp + "\n" +
                  nonce + "\n" +
                  sha256_hex(body);

String signature = hmac_sha256_base64url(device_secret, material);
```

Important details:

- Sign only the pathname and query, not scheme or hostname.
- Preserve the exact query ordering used in the HTTP request.
- For a GET request, hash an empty body.
- For a POST request, serialize JSON once, hash those exact bytes, then send the
  same `String`. Do not re-serialize after signing.
- Use lowercase SHA-256 hex.
- Use unpadded base64url for the HMAC and nonce: `-` and `_`, no `=` padding.
- The device clock must be synchronized before making a signed request.
- Portal permits at most 120 seconds of clock skew.

Do not change the QR-code signature contract. QR values continue to use:

```text
<device_id>.<Unix timestamp seconds>.<nonce>
```

That QR signature is separate from the five-line HTTP request signature.

## Recommended HTTP API

Replace the unsigned helpers with one signed request path. A suitable internal
interface is:

```cpp
struct HttpResult {
    int status_code = 0;
    String body;
    String error;
};

HttpResult signed_get(
    const DeviceConfig& config,
    const String& path_and_query);

HttpResult signed_post_json(
    const DeviceConfig& config,
    const String& path,
    const String& body);
```

The helper should:

1. Refuse to send until Wi-Fi and time synchronization are ready.
2. Generate timestamp and nonce.
3. Hash the exact body.
4. Build and HMAC the canonical material.
5. Add all four `X-PTC-*` headers.
6. Add `Content-Type: application/json` for POST requests.
7. Enforce connection and response timeouts.
8. Return HTTP status and response body without logging credentials, headers,
   QR payloads, manual codes, or signatures.

### TLS requirement

Remove `client.setInsecure()` for Portal requests. Configure
`WiFiClientSecure` with a trusted root CA or the ESP32 certificate bundle.
Trust a root CA, not the current leaf certificate, so routine certificate
renewal does not break the device.

The clock must be synchronized before TLS certificate validation.

### Do not block LVGL

Do not run network calls directly on the LVGL/UI loop. Use a FreeRTOS worker
task and a queue, following the existing registration-task pattern. Only the
main/LVGL thread may mutate LVGL objects.

Use one serialized request worker rather than launching overlapping config,
heartbeat, notice, and manual-code requests. Suggested priority:

1. Manual code for a newly generated QR payload.
2. Config refresh.
3. Heartbeat.
4. Notices.

## Route contracts

### 1. Config

```http
GET /api/timeclock/devices/config?device_id=cb9008f8-0098-4b46-b77b-b82029aff3f2
```

Successful response:

```json
{
  "device_id": "cb9008f8-0098-4b46-b77b-b82029aff3f2",
  "location_id": "9157afb6-fd8e-4a50-bcc6-7152b1e53669",
  "location_name": "Luton Office - Dunstable Road",
  "qr_interval_sec": 20,
  "is_active": true
}
```

Behavior:

- Fetch immediately after Wi-Fi and time sync are ready.
- Refresh every 5 minutes.
- Persist location, interval, and active state using `service_storage`.
- Stop displaying QR and manual codes while `is_active` is false.
- Do not erase the device ID or secret when the device is inactive.

### 2. Heartbeat

```http
POST /api/timeclock/devices/heartbeat
```

Request body:

```json
{
  "device_id": "cb9008f8-0098-4b46-b77b-b82029aff3f2",
  "firmware_version": "0.1.0",
  "ip": "192.0.2.10",
  "wifi_rssi": -55,
  "free_heap": 176000,
  "uptime_sec": 3600
}
```

Use:

```cpp
WiFi.localIP().toString();
WiFi.RSSI();
ESP.getFreeHeap();
millis() / 1000;
```

Successful response:

```json
{
  "ok": true,
  "server_time": 1784592000
}
```

Send every 60 seconds. `server_time` is Unix seconds, not an ISO string. It may
be used to detect clock drift, but normal SNTP remains the source of time.

### 3. Notices

```http
GET /api/timeclock/notices?device_id=cb9008f8-0098-4b46-b77b-b82029aff3f2
```

After the first successful response, the optional `since` value may be used:

```http
GET /api/timeclock/notices?device_id=<UUID>&since=<Unix seconds>
```

The response is a JSON array, not an object wrapper:

```json
[
  {
    "id": "<notice UUID>",
    "title": "Office notice",
    "body": "Notice text",
    "image_url": null,
    "hyperlink_url": null,
    "display_seconds": 6,
    "sort_order": 0,
    "created_at": "2026-07-21T00:00:00.000Z",
    "updated_at": "2026-07-21T00:00:00.000Z"
  }
]
```

Behavior:

- Fetch immediately after initial config.
- Refresh every 3 minutes and when the user taps the existing refresh action.
- An empty array is a valid successful response; clear stale cached notices.
- Update LVGL notice objects only on the main/UI thread.

### 4. Physical manual code

```http
POST /api/timeclock/devices/manual-code
```

Request body:

```json
{
  "device_id": "cb9008f8-0098-4b46-b77b-b82029aff3f2",
  "qr_payload": "ptc1:<the current signed QR payload>"
}
```

Successful response:

```json
{
  "code": "12345678",
  "code_display": "1234-5678",
  "expires_at": "2026-07-21T12:00:30.000Z"
}
```

Behavior:

- Request a code once whenever `service_qr_payload()` changes to a new non-empty
  value.
- Use that exact QR payload in the request body.
- Never attempt manual-code generation locally.
- Display `code_display`, not the unformatted `code` value.
- Clear the displayed code when it expires, the QR payload changes, the device
  is deactivated, Wi-Fi/time becomes invalid, or the request fails.
- Do not persist manual codes to Preferences, flash, SD, or logs.
- A new code replaces the previous unused code for this physical device.
- Portal rate-limits requests to the configured `qr_interval_sec`.

Expose service state to the QR UI, for example:

```cpp
String service_http_manual_code_display();
uint32_t service_http_manual_code_expires_at();
bool service_http_manual_code_pending();
```

In `src/ui/ui_qr.cpp`:

1. Add `lv_obj_t* code_value` to `QrUi`.
2. Assign the existing manual-code value label to `ui.code_value`.
3. Update it from the HTTP service state in the existing one-second timer.
4. Show `Unavailable` when no valid code exists and `Loading...` only while a
   request is pending.

## Scheduling changes

Remove the provisioned-device early return in `service_http_tick()`:

```cpp
if (state.provisioning_complete) {
    g_api_ok = api_endpoint_configured();
    g_last_error = g_api_ok ? "" : "API endpoint is not configured";
    return;
}
```

Replace it with scheduling logic for provisioned devices. The function should
remain fast and only enqueue work or consume completed worker results.

Suggested state:

```cpp
uint32_t g_last_config_ms = 0;
uint32_t g_last_notice_ms = 0;
uint32_t g_last_heartbeat_ms = 0;
uint32_t g_last_manual_qr_refresh_ms = 0;
```

Use wrap-safe elapsed comparisons:

```cpp
if (millis() - last_ms >= interval_ms) {
    // enqueue request
}
```

Do not use `millis() >= next_time` for long-running recurring schedules because
`millis()` wraps.

Registration behavior for `ims.piyamtravel.com` must remain disabled. A
provisioned production device uses the administrator-created UUID and secret;
it must not call `/api/timeclock/devices/register`.

## HTTP status handling

Handle statuses consistently:

| Status    | Firmware behavior                                                             |
| --------- | ----------------------------------------------------------------------------- |
| `200`     | Parse and apply response.                                                     |
| `400`     | Log a short redacted protocol error; do not tight-loop retry.                 |
| `401`     | Discard nonce, resync time, then retry later with a new nonce.                |
| `403`     | Mark device inactive and stop QR/manual-code display.                         |
| `429`     | Respect `retry_after` when present; do not request another manual code early. |
| `500-599` | Keep cached config/notices and use capped exponential backoff.                |

Never retry with the same timestamp, nonce, body signature, or signed header
set.

## Logging and secret rules

Permitted log data:

- Route name.
- HTTP status.
- High-level result such as `heartbeat sent` or `config rejected`.
- Retry delay.

Never log:

- Device secret.
- `X-PTC-Signature`.
- Request nonce.
- Full QR payload.
- Manual code.
- Full signed request headers or body.

Do not include any of those values in crash reports or screenshots.

## Build and test sequence

1. Update local `main` and confirm baseline `329fcfa` or newer.
2. Add the rotated secret to ignored `local.env`.
3. Build:

   ```bash
   pio run
   ```

4. Run once with serial monitoring and confirm no credentials are printed.
5. Confirm Wi-Fi and SNTP are ready before the first signed request.
6. Confirm config returns the Luton location and 20-second interval.
7. Confirm heartbeat updates firmware version, IP, RSSI, heap, uptime, and
   online state in PT Portal.
8. Confirm notices parse as an array and an empty array clears stale notices.
9. Confirm each new QR receives one 8-digit manual code and the old code stops
   working after replacement/expiry/use.
10. Confirm an authenticated employee can submit the displayed code once in PT
    Portal.
11. Replay the same signed request in a controlled test and confirm Portal
    returns `401`.
12. Change one signed body byte without recomputing the signature and confirm
    Portal returns `401`.
13. Temporarily test an incorrect device secret and confirm requests are
    rejected without exposing credentials.
14. Restore the correct secret and verify recovery uses a new nonce.
15. Confirm the LVGL UI remains responsive during DNS, TLS, timeout, and Portal
    failure cases.

## Definition of done

Firmware work is complete when all of these are true:

- QR generation remains compatible with `ptc1:` scanning.
- Every Portal service request uses the exact signed-header contract.
- TLS certificate validation is enabled.
- Provisioned devices actually run config, heartbeat, notice, and manual-code
  requests.
- HTTP work cannot block LVGL rendering/touch handling.
- Manual codes come only from Portal and are never persisted or logged.
- Config changes survive reboot.
- Inactive devices stop displaying valid QR/manual codes.
- The Portal device page reports online status and firmware `0.1.0` or the new
  release version.
- The firmware builds successfully and a physical-device smoke test passes.

After validation, increment `kFirmwareVersion`, publish a GitHub release with a
`firmware.bin` asset, and perform the first installation through USB or Arduino
OTA according to the repository README.
